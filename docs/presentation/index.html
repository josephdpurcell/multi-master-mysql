What data profiles
does this work with?

Solitary + Unique Indexes

Problem 1: Stale Data

  • IF: Read frequency is faster than replication latency.
  • THEN: The user will see stale data.
  • EXAMPLE: Update background color of your user profile.
    • T1: UPDATE user_profiles SET profile_bg_color='#FF0000' WHERE user_id=1
    • T2: SELECT * FROM user_profiles WHERE user_id=1
    • T3: replication completes

Solutions to Problem 1

  1. Use session fixation and last updated checks.
  • e.g. logged in on offline mobile device and on laptop

    • Problem 2: Duplicate Keys

    • IF: You have a unique index other than an auto increment
    • AND: Write frequency is faster than replication latency.
    • THEN: Then duplicate key errors will trigger.
    • EXAMPLE: Two people update their email at the same time.
      • T1: on node1: UPDATE users SET email='joe@example.com' WHERE id=1
      • T1: on node2: UPDATE users SET email='joe@example.com' WHERE id=1
      • T2: node1 receives the update from node2 and triggers dup error.
      • T2: node2 receives the update from node1 and triggers dup error.

    Solutions to Problem 2

    1. This should never happen.
    2. It's solitary data: make sure only one person is editing at a time.
    3. Use a write-back cache and do lookups to enforce uniqueness in the application layer.
    4. Database admins ftw! (Also, expensive licenses.)

    Solitary + No Unique Indexes

    This doesn't exist.

    Social + Unique Indexes

    Problem 1: Stale Data

    • IF: Read frequency is faster than replication latency.
    • THEN: The user will see stale data.
    • EXAMPLE: When will this happen?

    Problem 2: Duplicate Keys

    • IF: You have a unique index other than an auto increment
    • AND: Write frequency is faster than replication latency.
    • THEN: Then duplicate key errors will trigger.
    • EXAMPLE: Two people upload a spreadsheet of locations.
      • T1: on node1: INSERT INTO places (business_id, name) VALUES (1, 'Arbys on Michigan');
      • T1: on node2: INSERT INTO places (business_id, name) VALUES (1, 'Arbys on Michigan');
      • T2: node1 receives the Michigan loc from node2 and triggers dup error.
      • T2: node2 receives the Michigan loc from node1 and triggers dup error.

    Solutions to Problem 2

    1. Assume the data is write-only, e.g. FitBit data points.
    2. Make sure only one person is editing at a time.
    3. Use a write-back cache and do lookups to enforce uniqueness in the application layer.
    4. Database admins ftw! (Also, expensive licenses.)

    Social + No Unique Indexes

    There are no replication problems.
